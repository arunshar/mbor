MBOR, explained

A plain-language tour of Multi-level Bi-objective Routing and how this Rust reimplementation reproduces it. Companion to the project hub and the paper page.

0. The one mental model

Precompute the hard part between region boundaries once. Then any trip is "get to the edge of your region, cross a tiny graph of region-borders, arrive from the edge of the destination region", and you only ever combine a handful of precomputed Pareto sets.

That is the whole idea. Everything below is how you make that exact and fast when each road has two costs and there can be many equally-good answers.

1. The problem: bi-objective routing

Each edge carries two non-negative costs, say travel time and energy. A path's cost is the pair-wise sum. Path p dominates p' if it is no worse on both costs and strictly better on at least one. The answer to a query is the Pareto frontier: every path that nobody dominates. There is no single "shortest" path, there is a trade-off curve, and a driver picks a point on it (eco vs fast).

This is hard: the number of Pareto-optimal paths can grow exponentially, so the classic compute-on-demand searches (bi-objective Dijkstra, NAMOA*dr, BOA*) get slow on city-scale graphs. The paper's toy (Figure 1) has frontier n0 -> n7 = {(10,17), (11,16)}: two incomparable routes, one a little faster, one a little lighter.

2. How MBOR works

MBOR trades precomputation for query speed using three ideas.

Partition into fragments and find the boundary nodes

Cut the map into k fragments. A boundary node is any node with an edge crossing into another fragment. Every cross-region trip must pass through boundary nodes, so they are the "border crossings" of the map.

FPPV: fragment Pareto path views

Inside each fragment, precompute the Pareto frontier from every node to each of its boundary nodes, and from each boundary node to every node. That is the "get to the edge / arrive from the edge" part, computed once per fragment with the label-setting search restricted to that fragment.

Boundary multigraph + BPPV

Build a small graph over only the boundary nodes. Between two boundary nodes in the same fragment there can be several incomparable within-fragment routes, so the edge is a multi-edge carrying a whole Pareto cost-set; cross-fragment original edges are added too. The BPPV is the Pareto frontier between boundary-node pairs over this multigraph. Together, FPPV + BPPV form the MEPFV (the Multi-level Encoded Pareto Frontier View).

Online retrieval

MBOR-Basic (Alg 3): for a query (o, d), combine, over boundary-node pairs (oBN, dBN), the three precomputed Pareto sets FPPV(o→oBN) ⊕ BPPV(oBN→dBN) ⊕ FPPV(dBN→d) with the Minkowski sum (pairwise add, keep non-dominated), plus any path that stays inside one fragment.

MBOR-Adv (Alg 4): the combination above is the bottleneck because many boundary-node pairs cannot possibly contribute. Each pair's best conceivable result is its ideal corner (min total c1, min total c2). If that corner is already dominated by the frontier found so far, every path through that pair is dominated too, so its Minkowski sum is skipped. Processing pairs in ideal-corner order makes the frontier tighten early and prune the rest. This is the paper's two-dimensional cost-interval pruning, and it returns the identical exact frontier as Basic.

3. Design decisions (what, why, alternative)

DecisionWhyAlternative cost
Precompute MEPFV between boundariesTurns a city-scale search into combining a few small Pareto setsPure compute-on-demand (BOA*/NAMOA*dr) is exact but slow at scale
Multi-edges in the boundary graphTwo boundary nodes can have several incomparable connecting routes; a single edge would lose Pareto optionsAn ordinary boundary graph drops valid trade-offs (wrong answer)
2D cost-interval pruning (Adv)Skips boundary-pair combinations whose best case is already beaten; the Minkowski sum is the expensive stepBasic does every combination; correct but slower

4. The code (Rust workspace)

  • mbor-core: CSR graph + DIMACS loader, the bi-objective label-setting search (Alg 2 / BOA* family), Pareto dominance, Minkowski sum, and a one-to-all variant used to build the FPPV.
  • mbor-precompute: partition (BFS region-growing, or load a KaHIP file), boundary detection, FPPV via forward and reverse label-setting on induced fragment subgraphs, the boundary multigraph, BPPV via multigraph label-setting, and the online query (Basic) and query_adv (Adv, with pruning stats).
  • mbor-baseline: exact BOA* and bi-objective Dijkstra (BOD) for the speedup comparison.
  • mbor-bench: times all four methods on the BAY subnetworks, verifies they agree, and reports the speedups and pruning rate.

Correctness is enforced by tests: query and query_adv are checked against the exact full-graph search and both baselines for every node pair on the paper toy and on synthetic grids across many partition counts.

5. How to read the numbers

On 1/20 BAY (15,366 nodes, 50 real queries): BOA* 5.59 ms, MBOR-Basic 0.62 ms, MBOR-Adv 0.054 ms per query; MBOR-Adv is 104× faster than BOA*, and its 2D cost-interval pruning skips 94% of boundary-pair combinations. Zero exactness mismatches: the four methods return the same frontier. The paper reports MBOR-Basic 0.64 ms and a ">10×" headline for the same network; both are reproduced. See the hub for the full table and the evidence file.

6. Honest scope

Rust is not faster than good C++. Same compiler back end. The point of the Rust port is a memory-safe, one-command, reproducible artifact that reproduces the paper on open data, not a language speed-up.
Precompute is correct but not memory-optimized. It stores the full fragment Pareto view (per boundary node, a Pareto set to every node in its fragment), which is fine for the BAY subnetworks but heavy at full-BAY scale. The reproduced figures are online-query numbers; the paper's published full-BAY figures are cited separately, not claimed here.
Partition. The reproduced numbers use a dependency-free BFS region-growing partition. A KaHIP min-cut partition (the paper's choice) makes fewer boundary nodes and would speed MBOR further, so these numbers are conservative.

7. What is next
